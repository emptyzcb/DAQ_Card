# 信息采集板卡软件设计规划

## 1. 文档目的

本文档用于梳理信息采集板卡的软件部分需要实现的功能模块、当前工程已有基础，以及后续开发优先级。

当前工程基于 STM32H743，已经具备 FreeRTOS、USART、SPI、IMU 采集、DataHub 数据中心、USART3 DMA + IDLE + RingBuf 接收测试等基础能力。

## 2. 软件总体目标

信息采集板卡的软件核心目标是：

```text
稳定采集
统一缓存
可靠通信
可配置
可诊断
可升级
```

软件需要把底层传感器、模拟量、数字量、通信接口等硬件资源组织起来，形成稳定的数据采集和通信系统。

## 3. 推荐软件分层

```text
应用层
  |
服务层 Services
  |
BSP 板级驱动层
  |
HAL / LL 外设驱动层
  |
硬件外设
```

各层职责如下：

| 层级 | 作用 |
|---|---|
| 应用层 | 创建任务、组织业务流程、管理系统状态 |
| 服务层 | 采集服务、通信服务、数据处理、参数管理 |
| BSP 层 | 封装具体板卡硬件接口 |
| HAL/LL 层 | STM32 外设底层驱动 |
| 硬件层 | MCU、传感器、电源、通信接口、采集通道 |

## 4. 软件功能模块规划

| 模块 | 软件要做什么 | 典型内容 | 当前工程状态 | 建议优先级 |
|---|---|---|---|---|
| 底层驱动 | 把 MCU 外设驱动起来 | GPIO、UART、SPI、I2C、ADC、TIM、DMA、CAN、USB、Ethernet | 已有 USART2、USART3、SPI1、DMA 接收测试 | 高 |
| BSP 板级层 | 封装板卡硬件接口 | `bsp_led`、`bsp_console`、`bsp_rs485`、`bsp_adc`、`bsp_flash` | 已有 `bsp_led`、`bsp_console`、`bsp_imu660rc` | 高 |
| RS485 接收 | 实现可靠串口接收 | DMA 环形缓冲、半满中断、全满中断、IDLE 空闲中断、RingBuf | 已实现 USART3 DMA + IDLE + RingBuf | 高 |
| RS485 发送 | 实现 RS485 半双工发送 | DE/RE 方向控制、发送完成等待 TC、切回接收 | 当前只是 TTL 串口 echo，未加 DE/RE | 高 |
| 通信协议 | 定义上位机和板卡的数据格式 | 帧头、地址、命令、长度、payload、CRC16、应答、超时 | 已有 `stream_protocol` 雏形，但还没有完整命令解析 | 高 |
| 采集服务 | 周期读取传感器和输入信号 | IMU、ADC、DI、DO、温度、脉冲、编码器 | 已有 IMU/姿态相关服务 | 高 |
| DataHub 数据中心 | 统一保存最新采集数据 | IMU 数据、ADC 数据、DI 状态、系统状态、错误码 | 已有 IMU DataHub | 高 |
| 参数管理 | 保存和修改板卡配置 | 设备地址、波特率、采样周期、滤波参数、校准参数、报警阈值 | 暂未完整实现 | 中 |
| 数据处理 | 对采集数据做换算和滤波 | 单位换算、均值滤波、中值滤波、一阶低通、校准补偿 | IMU 姿态解算已有基础 | 中 |
| 任务管理 | 用 FreeRTOS 管理功能模块 | 采集任务、通信任务、LED 任务、监控任务、日志任务 | 已有姿态任务、串口 echo 测试任务 | 高 |
| 错误诊断 | 记录系统异常和通信异常 | CRC 错误、超时、RingBuf 溢出、传感器异常、电源异常 | 已有 drop 计数，其他还需扩展 | 中 |
| 状态指示 | 通过 LED 或日志显示运行状态 | 运行灯、通信灯、错误灯、心跳日志 | 已有 LED blink | 中 |
| 存储功能 | 保存参数、日志和采集记录 | Flash、EEPROM、FRAM、SD 卡、文件系统 | 暂未完整实现 | 中 |
| Bootloader/升级 | 支持现场固件升级 | 串口 Ymodem、RS485 升级、USB 升级、固件 CRC、回滚 | 参考工程里有 OTA，可后续移植 | 低到中 |
| 上位机测试 | 辅助调试和验证协议 | 串口助手、Python 脚本、CSV 记录、参数配置工具 | 当前用串口助手测试 | 中 |
| 系统监控 | 保证程序长期稳定运行 | 看门狗、任务心跳、堆栈监控、复位原因记录 | 暂未完整实现 | 中 |
| 产测功能 | 方便生产测试板卡 | IO 自检、通信自检、传感器 ID 检测、版本读取 | 暂未实现 | 低 |
| 文档 | 方便调试、维护和交付 | 通信协议文档、寄存器表、接线说明、测试流程 | 正在整理 | 中 |

## 5. 当前 USART3 接收测试结构

当前 USART3 用于验证串口接收链路，暂时按 TTL 串口方式测试，还没有使用 RS485 DE/RE 方向控制。

```text
USART3_RX
  |
DMA 环形接收缓冲区
  |
DMA 半满中断 / DMA 全满中断 / USART IDLE 空闲中断
  |
Pump 搬运新增数据
  |
软件 RingBuf
  |
测试线程读取并通过 USART3_TX 回显
```

当前测试特性：

| 项目 | 当前设置 |
|---|---|
| 串口 | USART3 |
| TX | PB10 |
| RX | PB11 |
| 波特率 | 115200 |
| 数据格式 | 8N1 |
| DMA RX buffer | 10 字节，便于测试半满和全满 |
| RingBuf | 2048 字节 |
| 半满触发 | 收到 5 字节 |
| 全满触发 | 收到 10 字节 |
| 空闲触发 | 收到不足半包并停止发送一段时间 |
| 调试日志 | `printf` 已改回 USART2 |

## 6. 后续开发路线

建议按以下顺序推进：

| 阶段 | 目标 | 主要工作 |
|---|---|---|
| 1 | 完成 USART3 接收稳定性验证 | 测试 DMA 半满、全满、IDLE、RingBuf、drop 计数 |
| 2 | 封装 BSP_RS485 | 增加 DE/RE 控制、发送完成等待 TC、切回接收 |
| 3 | 建立通信协议 | 定义帧头、地址、命令、长度、payload、CRC16 |
| 4 | 接入 DataHub | 从 DataHub 读取 IMU/姿态数据并通过协议返回 |
| 5 | 增加参数系统 | 保存设备地址、波特率、采样周期、校准参数 |
| 6 | 扩展采集通道 | 增加 ADC、DI、DO、温度、脉冲等采集服务 |
| 7 | 完善诊断能力 | 增加错误码、任务心跳、通信错误计数、看门狗 |
| 8 | 增加升级能力 | 移植 Ymodem 或自定义 RS485 升级协议 |

## 7. 最小可用软件版本

第一版建议实现以下能力：

```text
1. USART2 调试日志
2. USART3 RS485 接收和发送
3. DMA + IDLE + RingBuf 接收
4. RS485 DE/RE 半双工控制
5. 简单二进制协议
6. 查询设备信息
7. 查询实时 IMU/姿态数据
8. 查询错误状态
9. LED 运行状态指示
10. 基础参数保存
```

## 8. 注意事项

1. H743 的 DMA 不能随便访问所有 RAM，DMA buffer 应放在 DMA 可访问区域，例如 AXI SRAM。
2. 中断服务函数里不要直接 `printf`，应只做计数、置位或搬运轻量数据。
3. RS485 半双工发送时，必须等待 UART `TC` 标志置位后再切回接收。
4. RingBuf 底层建议保持简单，协议解析层可以单独维护解析状态机或 peek 指针。
5. CubeMX 重新生成代码后，要检查 USART3 DMA、NVIC、中断入口和 Keil 工程文件是否被覆盖。
